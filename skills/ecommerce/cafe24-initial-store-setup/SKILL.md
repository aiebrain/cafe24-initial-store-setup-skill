# Cafe24 초기 자사몰 설정

사용자가 Cafe24 관리자에 직접 로그인한 뒤 제공한 사업자·고객센터·운영 정보를 바탕으로, 한국어 쇼핑몰의 초기 설정을 입력·저장·공개 화면에서 검증한다.

## 범위와 안전 경계

- 먼저 대상 상점 ID/도메인을 확인한다. 동일 브라우저에 여러 Cafe24 상점이 열려 있으면 URL과 폼의 `mall_id`/상점 ID가 사용자가 지정한 대상과 일치할 때만 수정한다.
- 로그인, OTP, 본인인증, PG 가입/계약/서류제출은 사용자가 직접 수행한다. 비밀번호·쿠키·인증번호·계좌 비밀번호를 요청하거나 저장하지 않는다.
- 사업자등록증, 통신판매업 신고증 등 공식 증빙과 맞지 않는 사업자 정보는 추측·생성하지 않는다.
- 팩스번호, 회사 약도 이미지, 사업자용 계좌 등 제공되지 않은 선택 정보는 임의로 채우지 말고 `미입력(정보 미제공)`으로 보고한다.
- PG 신청/계약 버튼은 사용자의 별도 명시 지시 없이 누르지 않는다. 이 스킬은 PG **승인 전 준비 상태**까지를 기본 범위로 한다.

## 시작 전 사용자에게 받을 정보

필수:

1. 대상 Cafe24 상점 URL 또는 상점 ID
2. 쇼핑몰명 / 상호(법인명) / 대표자명
3. 사업자등록번호, 업태, 종목
4. 사업장 주소, 대표전화
5. 수신용 이메일, CS 전화·이메일, 실제 CS 운영시간
6. 통신판매업 신고 여부와 신고번호(발급 상태라면)
7. 개인정보보호책임자: 성명, 직위, 부서, 전화, 이메일
8. 회사소개 1~3문장 및 서비스 문의 안내 문구

선택:

- 대표·CS 팩스, SMS 수신번호, 회사 약도 이미지
- 대표 도메인/홈페이지 URL
- 무통장 은행·계좌번호·예금주(무통장 결제를 실제 제공할 때만)
- 배송·교환·반품 정책과 출고지

## 실행 절차

### 1. 브라우저 실행 경로 선택, 로그인과 대상 상점 확인

브라우저 하네스를 전제로 하지 않는다. 실행 가능 수단을 아래 우선순위로 확인하고, 가능한 수단만 사용한다.

1. **Windows Chrome CDP / browser-harness 사용 가능**: `browser-harness-wsl`을 로드한다. 새 탭으로 관리자 URL을 열어 기존 사용자의 가시적 Chrome에서 작업한다.
2. **Hermes `browser` 도구 사용 가능**: `browser_navigate`로 관리자 URL을 연다. 브라우저 세션에 로그인 화면이 나오면 사용자가 그 세션에서 로그인·OTP를 직접 완료할 때까지 대기한다. 로그인 전에는 공개 페이지 점검과 입력 계획만 수행한다.
3. **로컬 데스크톱 브라우저 + computer-use 사용 가능**: `computer-use`를 로드해 Chrome/Edge 등 사용자의 브라우저 창을 열고 주소창에 관리자 URL을 입력한다. 사용자가 로그인한 뒤 폼을 읽고 수정한다.
4. **위 수단이 모두 없으면 Playwright MCP 설치 후 진행**:
   - 먼저 `node`, `npx`, `hermes mcp list`로 Node.js와 기존 Playwright MCP 설정을 확인한다.
   - Playwright가 없다면 공식 Microsoft 패키지를 native stdio MCP로 설치한다. 명령은 `hermes mcp add playwright --command npx --args -y @playwright/mcp@latest --browser chromium`을 사용한다. 브라우저 배포판이 없으면 `npx -y @playwright/mcp@latest install-browser chrome-for-testing` 또는 `npx -y playwright install chromium`을 실행한다.
   - `hermes mcp test playwright` 뒤 `https://example.com`으로 실제 `browser_navigate` + `browser_snapshot` smoke test를 수행한다. 새 MCP 도구는 `/reload-mcp` 또는 새 Hermes 세션 후에 사용한다.
   - Cafe24 로그인에는 사용자가 직접 조작할 수 있는 **가시적(headed) 브라우저 표면**이 필요하다. 원격/헤드리스 환경에서 사용자에게 로그인 창이 보이지 않으면 로그인을 우회하지 않는다. 사용자가 Playwright가 연 브라우저에 접근할 수 있도록 하거나, 2·3번 경로로 전환한다.
5. **Playwright도 가시적 로그인 표면도 없는 환경**: 자동 수정은 수행하지 않는다. 사용자가 자신의 브라우저에서 열 수 있는 정확한 관리자 URL, 필드별 입력값, 저장 후 공개 푸터 점검표를 제공한다. 브라우저 도구/데스크톱 연결 설정을 안내할 수 있지만, 설치를 자동으로 요구하지 않는다.

공통 절차:

1. 대상 관리자 URL을 열고 사용자에게 로그인·인증을 맡긴다.
2. 로그인 후 URL이 다른 상점으로 리디렉션되지 않았는지 확인한다.
3. `내 쇼핑몰 정보` 화면의 상점 ID, 기본 제공 도메인, 쇼핑몰명을 읽고 대상과 대조한다.
4. 대상 상점이 확인되기 전에는 어떤 저장 버튼도 누르지 않는다.

대상 메뉴:

`쇼핑몰 설정 → 기본 설정 → 내 쇼핑몰 정보`

대표 URL 패턴:

`https://<mall-id>.cafe24.com/admin/php/shop1/m/company_info_f.php`

### 2. 기존값을 먼저 읽고 입력 계획 제시

- 저장 전 폼의 현재값을 읽어 사용자 제공값과 비교한다.
- 사업자 정보가 이미 채워졌으면 덮어쓰기 전에 차이를 명확히 보여준다.
- 사업자등록증과 다를 가능성이 있는 항목(상호, 대표자, 번호, 주소, 업태·종목)은 저장하지 말고 사용자에게 확인받는다.
- 법정 고지와 고객에게 공개되는 문구는 오탈자·오래된 업종 문구(예: 이전 쇼핑몰 소개)가 없는지 확인한다.

### 3. 내 쇼핑몰 정보 입력

`내 쇼핑몰 정보`에서 아래를 설정한다.

| 영역 | 입력 항목 | 푸터/PG 관련성 |
|---|---|---|
| 기본 | 쇼핑몰명, 관리자명 | 쇼핑몰 식별 |
| 사업자 정보 | 사업자등록번호, 상호, 대표자, 업태, 종목, 주소, 대표전화, 수신용 이메일, 홈페이지, 통신판매업 신고번호 | 푸터·매출전표·PG 심사 핵심 |
| 회사소개 | 제공된 실제 사업/서비스 설명 | 회사소개 링크 |
| 고객센터 | 상담/주문 전화, 이메일, SMS 수신번호, 실제 운영시간 | 푸터 노출 |
| 개인정보보호책임자 | 이름, 직위, 부서, 전화, 이메일 | 푸터·개인정보 고지 |
| 서비스 문의 | 모바일 표시 `표시함`, 제공된 문의 안내 | 모바일 안내 |

권장 문구 원칙:

- 회사소개: 판매/교육/디지털 서비스 등 실제 사업 범위를 1~3문장으로 명확히 쓴다.
- CS운영시간: `24시간 운영`이라고 쓰지 않는다. 주문 접수와 상담 응대가 다르면 구분한다. 예: `주문은 24시간 접수됩니다. 상담 문의는 영업일 기준 순차적으로 답변드립니다.`
- 서비스 문의: 고객이 실제 사용할 전화·이메일과 응답 기준을 적는다.

저장 후 페이지를 새로고침하고 입력한 값이 유지되는지 읽어 검증한다.

### 4. 이용약관 및 개인정보 안내

대상 메뉴:

`쇼핑몰 설정 → 기본 설정 → 약관·정보 설정 → 이용 약관 설정`

대표 URL 패턴:

`https://<mall-id>.cafe24.com/admin/php/shop1/m/mall_agree_f.php`

1. 기존 Cafe24 표준 양식을 먼저 읽는다.
2. 사용자가 요청한 경우에만 기존 양식을 유지한 채 상호(법인명)·쇼핑몰명이 실제 값으로 치환되어 있는지 확인한다.
3. 다른 사업자명, 이전 업종 문구, `{회사명}` 같은 미치환 자리표시자가 남아 있으면 수정한다.
4. 약관 전체를 임의로 새로 쓰지 않는다. 법률 자문이 필요한 조항이나 사용자 제공 정책이 없는 조항은 표준 양식을 유지한다.
5. 저장 후 새로고침하여 사업자명/쇼핑몰명 치환, 빈 자리표시자 부재, 이전 브랜드 문구 부재를 검증한다.

### 5. 무통장 결제 정보 확인

메뉴:

`쇼핑몰 설정 → 결제 설정 → 결제 정책 설정 → 무통장입금 계좌 설정`

- 사용자가 무통장 결제를 제공한다고 명시한 경우에만, 은행·계좌번호·예금주가 제공값과 일치하는지 확인한다.
- 푸터에 계좌번호가 보인다고 해서 주문서 결제수단에 연결된 것으로 간주하지 않는다.
- 최소 검증 3점: 관리자 무통장 계좌 설정 = 주문서 무통장 선택 화면 = 푸터 결제정보.
- 계좌 변경·등록은 사용자가 명시적으로 제공한 은행/계좌/예금주와 저장 지시를 준 경우에만 수행한다.

### 6. PG 승인 전 준비 항목 검증

PG 신청 자체는 하지 않는다. 아래만 점검한다.

1. 공개 도메인/기본 도메인에서 홈과 상품 상세가 열림
2. 실제 판매 상품이 최소 1개 이상 공개되어 있고 상품명·판매가·이미지·상세설명·배송 조건이 있음
3. 사업자 정보와 고객센터 정보가 푸터에 노출됨
4. 이용약관, 개인정보처리방침, 이용안내 링크가 열림
5. 배송비·출고지·취소/교환/반품 정책이 실제 운영 기준과 일치
6. 무통장 제공 시 계좌정보 3점 일치

### 7. 공개 쇼핑몰 검증

`https://<mall-id>.cafe24.com/`의 PC와 모바일 푸터에서 아래를 확인한다.

- 상호, 대표자, 주소, 대표전화
- 사업자등록번호, 통신판매업 신고번호, 사업자정보확인 링크
- 개인정보보호책임자
- 고객센터 전화, 이메일, 운영시간
- 이용약관, 개인정보처리방침, 이용안내 링크
- 무통장 제공 시 은행·계좌번호·예금주

푸터가 접힘(아코디언) 형태이면 DOM에서 실제 노출 데이터도 확인한다. 화면에 보이는 정적 푸터 값과 관리자/주문서 결제 설정을 혼동하지 않는다.

## 기술적 실행 팁

- 일반 브라우저 하네스가 다른 탭을 읽거나 `Runtime.evaluate`가 반복 타임아웃되면, `browser-harness-wsl`의 raw-CDP 복구 절차를 사용한다.
- Cafe24 `내 쇼핑몰 정보`는 일반적으로 `input[name=company_name]`, `input[name=president_name]`, `textarea[name=cs_run_time]`, `textarea[name=service_info]` 같은 필드를 포함한다. 실제 DOM을 읽어 확인한 뒤에만 사용한다.
- 회사소개는 Froala 편집기일 수 있다. `textarea[name=mall_info]`만 바꾸지 말고 편집기 HTML을 동기화한 뒤 저장한다.
- 저장 후에는 반드시 페이지 새로고침(read-back)으로 영속 상태를 확인한다.

## 완료 보고 형식

다음을 간결하게 표로 보고한다.

| 구분 | 결과 | 검증 |
|---|---|---|
| 사업자/푸터 정보 | 완료·보류 | 관리자 read-back + 공개 푸터 |
| 고객센터/개인정보책임자 | 완료·보류 | 관리자 read-back + 공개 푸터 |
| 회사소개/서비스 문의 | 완료·보류 | 관리자 read-back |
| 이용약관 표준양식 사업자명 | 완료·기존 완료·보류 | 약관 read-back |
| 무통장 계좌 | 확인됨·미확인·미사용 | 관리자/주문서/푸터 여부 |
| PG 승인 전 점검 | 통과·보완 필요 | 누락 항목 명시 |

변경하지 않은 선택 항목은 이유와 함께 명시한다. PG 신청·결제계약·개인 인증을 수행하지 않았음을 분명히 적는다.
